## Context

Phase 3 framework (network, operations, cluster, storage) is complete. All seven domain services exist as stubs generated by the `domain_stub!` macro, returning `OperationResponse::NotImplemented`. TODO-084 is the first Phase 3b task: replace the `CoordinationService` stub with real business logic.

The primary value of this spec is **validating the full message pipeline end-to-end**: WebSocket binary message -> MsgPack deserialize -> `OperationService::classify()` -> `OperationRouter` dispatch -> `CoordinationService` handler -> serialize response -> WebSocket binary out. Once this path works for three simple operations (Ping, PartitionMapRequest, heartbeat side-effect), all subsequent domain services (CRDT, Sync, Messaging, etc.) follow the same pattern.

**Scope boundary:** This spec implements only the `CoordinationService` domain logic. If the end-to-end pipeline (classify -> route -> handle -> respond) fails, the fix belongs in a separate pipeline wiring spec, not here. This service is testable in isolation via direct `tower::Service::call()` invocations without a live WebSocket.

**TS Source Reference:**
- `packages/server/src/coordinator/heartbeat-handler.ts` -- `handlePing()`: echoes client timestamp as `PONG`, updates `lastPingReceived`
- `packages/server/src/coordinator/partition-handler.ts` -- `handlePartitionMapRequest()`: compares client version, returns current partition map if stale

## Goal

Replace the `CoordinationService` domain stub with a real `tower::Service<Operation>` implementation that handles three coordination operation variants: `Ping`, `PartitionMapRequest`, and `LockRequest`/`LockRelease` (remaining as `NotImplemented` stubs for now).

### Observable Truths

1. A `Message::Ping(PingData { timestamp: T })` classified and routed to `CoordinationService` produces `OperationResponse::Message(Box::new(Message::Pong(PongData { timestamp: T, server_time: NOW })))`.
2. A `Message::PartitionMapRequest { payload: Some(PartitionMapRequestPayload { current_version: Some(V) }) }` where `V < table.version()` produces `OperationResponse::Message(Box::new(Message::PartitionMap { payload }))` containing the current partition map.
3. A `Message::PartitionMapRequest` where `current_version >= table.version()` produces `OperationResponse::Empty` (client already up-to-date).
4. A `Message::PartitionMapRequest { payload: None }` returns the full partition map when the partition table has been assigned (version > 0). When both the client version and table version are 0, `Empty` is returned (see Assumptions — version 0 vs 0 case).
5. When a `Ping` operation is handled, the `ConnectionMetadata.last_heartbeat` for the caller's connection is updated to `Instant::now()`.
6. `LockRequest` and `LockRelease` operations still return `OperationResponse::NotImplemented` (deferred to a future spec).

## Task

### 1. Create `CoordinationService` struct (file: `packages/server-rust/src/service/domain/coordination.rs`)

Define a concrete `CoordinationService` struct that replaces the `domain_stub!` generated version:

```rust
pub struct CoordinationService {
    cluster_state: Arc<ClusterState>,
    connection_registry: Arc<ConnectionRegistry>,
}
```

**Constructor:** `CoordinationService::new(cluster_state: Arc<ClusterState>, connection_registry: Arc<ConnectionRegistry>) -> Self`

**`ManagedService` implementation:**
- `name()` returns `service_names::COORDINATION` (same as the stub)
- `init()`, `reset()`, `shutdown()` are no-ops (return `Ok(())`)

**`tower::Service<Operation>` implementation for `Arc<CoordinationService>`:**

Match on the `Operation` variant:

- **`Operation::Ping { ctx, payload }`:**
  1. Compute `server_time` as milliseconds since epoch using: `SystemTime::now().duration_since(SystemTime::UNIX_EPOCH).unwrap_or_default().as_millis() as u64`. The `as u64` truncation from `u128` is safe for the next ~500 million years. Use `u64` here since `PongData.server_time` is `u64` (this differs from `to_partition_map()` in `state.rs` which casts to `i64` for `generated_at`).
  2. If `ctx.connection_id` is `Some(id)`, call `self.connection_registry.get(id)` to retrieve the `ConnectionHandle`. If the handle is found, acquire the write lock on `ConnectionMetadata` via `.write().await` (the field is behind `tokio::sync::RwLock`) and update `last_heartbeat` to `Instant::now()`. If `ctx.connection_id` is `None` or the handle is not found, skip the heartbeat update silently.
  3. Return `OperationResponse::Message(Box::new(Message::Pong(PongData { timestamp: payload.timestamp, server_time })))`

- **`Operation::PartitionMapRequest { ctx, payload }`:**
  1. Read the current `MembersView` from `cluster_state.current_view()`
  2. Read the current table version from `cluster_state.partition_table.version()` (returns `u64`)
  3. Extract `client_version` from `payload`: `payload.as_ref().and_then(|p| p.current_version).map(u64::from).unwrap_or(0)`. The `u64::from(v)` cast widens safely from `u32` (the wire type of `current_version`) to `u64` (the table version type), consistent with how `to_partition_map` already handles the reverse truncation.
  4. If `client_version < table_version`, call `cluster_state.partition_table.to_partition_map(&members_view)` and return `OperationResponse::Message(Box::new(Message::PartitionMap { payload: map }))`
  5. If `client_version >= table_version`, return `OperationResponse::Empty`

- **`Operation::LockRequest { ctx, .. }` | `Operation::LockRelease { ctx, .. }`:**
  Return `OperationResponse::NotImplemented { service_name: service_names::COORDINATION, call_id: ctx.call_id }`

- **Any other variant:**
  Return `Err(OperationError::WrongService)`

### 3. Heartbeat update via `ConnectionRegistry`

The `ConnectionRegistry` currently does not support lookup by `client_id` (string). The `ConnectionMetadata` has no `client_id` field either -- client identity is tracked externally.

**Design decision:** Rather than adding a `client_id` index to `ConnectionRegistry` (premature optimization), the `CoordinationService` will receive an optional `Arc<ConnectionHandle>` through the `OperationContext`. Since `OperationContext` currently has `client_id: Option<String>` but no handle reference, and adding `Arc<ConnectionHandle>` to `OperationContext` would change the operation pipeline significantly:

**Pragmatic approach:** Add a `connection_id: Option<ConnectionId>` field to `OperationContext`. The WebSocket handler (which knows the `ConnectionId`) sets this when creating the operation context. `CoordinationService` uses `connection_registry.get(connection_id)` to look up the handle and update heartbeat.

Changes to `OperationContext` (file: `packages/server-rust/src/service/operation.rs`):
- Add field `pub connection_id: Option<ConnectionId>` (default `None`)
- Import `ConnectionId` from `crate::network::connection`
- Update the existing doc comment on `OperationContext` (lines 49-52 of `operation.rs`) that states it "does not derive `Default` because `call_id`, `service_name`, and `timestamp` are required fields." Extend the doc comment to list `connection_id` among the optional fields alongside `partition_id`, `client_id`, and `caller_node_id`.

This is a minimal, backward-compatible change -- all existing code that constructs `OperationContext` via `new()` gets `None` by default. The WebSocket inbound handler will set it in a future spec when it integrates with `OperationService`.

## Acceptance Criteria

1. **Ping -> Pong:** `CoordinationService` receives `Operation::Ping { payload: PingData { timestamp: 42 } }` and returns `OperationResponse::Message` containing `Message::Pong(PongData { timestamp: 42, server_time: _ })` where `server_time` is within 1 second of `SystemTime::now()`.

2. **PartitionMapRequest (stale version):** Given a partition table with version 3, a request with `current_version: Some(1)` returns `OperationResponse::Message` containing `Message::PartitionMap { payload }` with `payload.version == 3`.

3. **PartitionMapRequest (current version):** Given a partition table with version 3, a request with `current_version: Some(3)` returns `OperationResponse::Empty`.

4. **PartitionMapRequest (no payload):** A request with `payload: None` returns the full partition map when the partition table has version > 0 (use a table with at least version 1 in this test). The version-0 edge case is documented in Assumptions.

5. **Heartbeat side-effect:** When handling a `Ping` operation with `connection_id: Some(id)`, the `ConnectionMetadata.last_heartbeat` for that connection is updated. Verified by reading the metadata before and after the operation and asserting `last_heartbeat` changed.

6. **Lock operations (not implemented):** `LockRequest` and `LockRelease` return `OperationResponse::NotImplemented`.

7. **Wrong service rejection:** A non-coordination operation (e.g., `Operation::GarbageCollect` with `service_name: "coordination"`) dispatched to the service returns `Err(OperationError::WrongService)`.

8. **ManagedService lifecycle:** `CoordinationService` registers in `ServiceRegistry`, `init_all` and `shutdown_all` succeed, and it is accessible via both `get_by_name("coordination")` and `get::<CoordinationService>()`.

9. **OperationContext backward compatibility:** All existing tests that construct `OperationContext::new()` compile and pass without modification (new field defaults to `None`).

10. **Clippy clean:** `cargo clippy --all-targets --all-features -- -D warnings` passes for `server-rust`.

11. **All existing tests pass:** `cargo test -p topgun-server` passes with no regressions.

## Assumptions

- **No HeartbeatAck wire message:** The task description mentions "HeartbeatAck" but no such message type exists in the wire protocol. The heartbeat acknowledgment is the `Pong` response plus the side-effect of updating `ConnectionMetadata.last_heartbeat`. This spec implements both.
- **Partition map version 0 treated as stale:** When the partition table has version 0 (freshly created, no assignments), a request with `current_version: None` or `Some(0)` should still return the (empty) map for consistency. The version comparison uses `client_version < table_version`, so version 0 vs 0 returns `Empty`.
- **`SystemTime` for server_time:** The `PongData.server_time` field uses wall-clock time (ms since epoch) from `SystemTime::now()`, not HLC time, matching the TS implementation (`Date.now()`).
- **No authentication check:** The TS `handlePing` checks `client.isAuthenticated` before evicting. This spec does not add authentication gating; the operation pipeline will gate on auth at the middleware layer in a future spec.
