# TopGun: Концепция продукта

## Одна строка

TopGun — offline-first база данных реального времени, которая даёт zero-latency чтение/запись на клиенте, бесконфликтную синхронизацию через CRDT и полноценный SQL — из коробки, с минимальной настройкой (`docker compose up` или TopGun Cloud).

---

## Проблема, которую мы решаем

Разработчик хочет добавить real-time в своё приложение. Сегодня у него четыре плохих варианта:

**Просто, но ограниченно.** Firebase даёт быстрый старт, но запирает в ограниченную модель данных. С 2023 года Firestore добавил OR-запросы, множественные inequality-фильтры и агрегации — для простого CRUD этого достаточно. Но каждая новая комбинация фильтров требует предсозданного composite index (лимит — 200 на базу), нет ad-hoc запросов, нет LIKE/regex. Полнотекстовый поиск по-прежнему отсутствует — официальная документация рекомендует Algolia или Typesense как отдельный сервис. JOINs нет — только денормализация с ручным поддержанием консистентности. Для e-commerce с поиском, фильтрацией и реляциями между сущностями — по-прежнему болезненно. Offline-поддержка формально есть (100MB кэш по умолчанию, настраивается), но модель слабая: last-write-wins без CRDT, нет selective sync, нет delta sync. При росте — vendor lock-in (нет стандартного API, нет self-hosting) и непредсказуемый счёт (read-based billing, fan-out listener costs).

**Мощно, но сложно.** Supabase предлагает PostgreSQL с real-time слоем, но это event-система (broadcast + presence + CDC), а не sync engine — нет conflict resolution, нет локального состояния. Встроенной offline-поддержки нет — это самый upvoted запрос в GitHub Supabase с 2020 года, до сих пор не реализован. Supabase Realtime имеет документированные проблемы стабильности: зависания соединений, WAL buildup, replication lag. На Supabase Cloud начальная настройка проще (WAL сконфигурирован автоматически), но для self-hosted — ручная настройка `wal_level`, `max_replication_slots`, логического декодирования. PostGraphile страдает от официально признанной проблемы "грохочущего стада" — одно обновление порождает тысячи SQL-запросов для всех подписчиков; документация прямо говорит: подходит для "internal tooling", не для "internet scale". Hasura решает это polling-архитектурой (1M подписок), но с компромиссом по latency.

**Offline-first, но без серверной мощности.** PowerSync, ElectricSQL, Instant, Triplit — новая волна local-first инструментов. PowerSync наиболее зрелый: полноценный offline read/write с локальным SQLite, поддержка PostgreSQL/MongoDB/MySQL/SQL Server, Fortune 500 клиенты через JourneyApps. Но нет встроенного CRDT conflict resolution — разработчик сам определяет merge-логику. ElectricSQL пережил два пивота (перезапись в 2024, ребрендинг в "multi-agent AI" в 2026) и принципиально ограничен: **только read-path sync** — write-path разработчик реализует сам. Требует PostgreSQL logical replication. Instant (YC-backed) и Triplit растут, но оба без SQL, без полнотекстового поиска, без серверной кластеризации.

**Real-time, но ненадёжно.** Pusher официально не гарантирует доставку сообщений — их документация прямо говорит: "Pusher does not provide an absolute guarantee of message delivery." Любое событие при обрыве сети исчезает. Нет persistence, нет replay, нет history. Billing при fan-out: 1 сообщение для 1,000 подписчиков = 1,001 сообщение в счёте; чат с 5,000 пользователями при 1 msg/sec = ~8.6M messages/day. Ably решает проблему потери данных (exactly-once delivery, message history), но стоимость при масштабе остаётся высокой. GUN.js — solo-maintainer проект без корпоративного бэкинга; по-прежнему нет query engine (только graph traversal по ключам, issue #337 с 2017 года открыт), нет фильтрации/сортировки/пагинации; зафиксирована path traversal vulnerability в npm audit 2026.

Общая проблема: **разработчик вынужден выбирать между простотой, мощностью, надёжностью offline и стоимостью**. Ни одно решение не даёт всё четыре одновременно.

## Что делает TopGun

TopGun снимает этот выбор. Данные живут в памяти клиента — чтение и запись мгновенны, не ждут сервера. Конфликты разрешаются автоматически через CRDT (conflict-free replicated data types) с Hybrid Logical Clock — без ручного merge, без потерь. Когда сеть возвращается, дельта-синхронизация через Merkle tree передаёт только изменения.

### Что это значит на практике

| Сценарий | Существующие решения | TopGun |
|----------|---------------------|--------|
| Пользователь offline 2 часа, редактирует данные | Firebase: last-write-wins, возможна потеря. Pusher: данные потеряны. PowerSync: работает, но merge — на разработчике | Всё сохранено локально через CRDT, синхронизировано автоматически при reconnect без конфликтов |
| 5,000 подписчиков на один документ | PostGraphile: "thundering herd" — тысячи SQL-запросов. Pusher: ~5K messages в счёте за одно обновление | In-memory broadcast, база не участвует в hot path, billing предсказуем |
| Нужен полнотекстовый поиск + фильтрация | Firebase: невозможно без Algolia ($). Supabase: только на сервере. Instant/Triplit: нет FTS | SQL (DataFusion) + full-text search (Tantivy) — из SDK и админ-панели (выполняется на сервере) |
| Масштабирование с 10 до 10,000 пользователей | Firebase: рост счёта в 10-100x. ElectricSQL: только read-sync. GUN.js: падает | Кластерное горизонтальное масштабирование, архитектура не меняется |

## Для кого

### Основная аудитория: разработчики, которые хотят строить, а не настраивать

Frontend-разработчики, инди-хакеры и стартап-команды (2-10 инженеров), которые строят приложения с real-time элементами: чаты, канбан-доски, коллаборативные редакторы, CRM, POS-системы.

Они умеют писать код, но не хотят тратить недели на инфраструктуру. Им нужно:

- **Начать за 5 минут:** `npm install @topgunbuild/client` — и данные синхронизируются.
- **Не думать об offline:** приложение просто работает без сети, без специальной логики.
- **Не упереться в потолок:** когда проект вырастет, не переписывать бэкенд.

### Вторичная аудитория: команды с offline-critical требованиями

Field service, логистика, healthcare, ритейл — приложения, которые **обязаны** работать без интернета. Для них CRDT-синхронизация — не удобство, а бизнес-требование.

## Принцип продукта: сложное скрыто, но доступно

TopGun по умолчанию прост. Подключил SDK, указал хранилище — работает. Админ-панель показывает данные, метрики, подключения без необходимости лезть в терминал.

Когда проект растёт, открываются следующие уровни:

| Потребность | Что доступно |
|-------------|-------------|
| Данных стало много | SQL-запросы из SDK и админ-панели (серверное выполнение), индексы через Admin API |
| Нужна безопасность | RBAC — ограничение доступа по ролям и коллекциям (maps) |
| Один сервер не справляется | Кластерное горизонтальное масштабирование |
| Старые данные занимают место | Автоматический tiering: холодные данные уходят в PostgreSQL (S3 — в roadmap) |
| Нужны вебхуки, интеграции | Webhooks для внешних событий |
| Мониторинг нагрузки | Встроенная наблюдаемость: метрики, алерты |

Новичку система не кажется перегруженной. Опытному инженеру она даёт всё, что нужно, — без смены платформы.

## Конкурентная позиция

TopGun занимает уникальную позицию: **единственный open-source продукт, совмещающий полноценный offline-first (CRDT) с серверным SQL, полнотекстовым поиском и кластеризацией**.

|  | Offline-first | Conflict resolution | SQL / FTS | Кластеризация | Open-source |
|--|--------------|-------------------|-----------|--------------|-------------|
| **TopGun** | Да (CRDT) | Автоматический (HLC) | SQL (DataFusion) + FTS (Tantivy) | Да (271 партиция) | Apache 2.0 |
| Firebase | Слабый (LWW, нет CRDT) | Last-write-wins | Нет SQL, нет FTS | Managed only | Нет |
| Supabase | Нет | Нет | PostgreSQL | Managed only | Да |
| PowerSync | Да (SQLite) | На разработчике | SQLite (клиент) | Нет | Да |
| ElectricSQL | Только read-path | Нет (нет write sync) | Нет | Нет | Да |
| Instant | Да | Сервер-арбитр | Нет SQL, нет FTS | Нет | Частично |
| Triplit | Да | CRDT-подобный | Свой query language | Нет | Да |
| Ditto | Да (CRDT) | Автоматический | Нет SQL | Да (mesh) | Нет |
| Convex | Нет | Server-authoritative | Свой query language | Managed only | Нет |
| RxDB | Да | На разработчике | Нет SQL | Нет | Premium |
| GUN.js | Концептуально | Нет (broken at scale) | Нет | Нет | Да |

Ближайшие конкуренты по offline-first (PowerSync, Triplit, Instant) не имеют серверного SQL и FTS. Ближайшие по серверной мощности (Supabase, Convex) не имеют offline-first. TopGun — единственный, кто совмещает оба.

### Что TopGun НЕ заменяет

TopGun — база данных реального времени, не full-stack платформа. Он не включает:

- **Аутентификацию** — используйте Clerk, Auth0, BetterAuth или любой JWT-совместимый провайдер. TopGun принимает стандартные JWT-токены.
- **Хостинг фронтенда** — используйте Vercel, Netlify, Cloudflare Pages. TopGun — это бэкенд.
- **Серверные функции** — нет аналога Cloud Functions. Для серверной логики используйте свой backend + webhooks.
- **Аналитику и ML** — нет встроенной аналитики. Используйте PostHog, Mixpanel или свои инструменты.
- **Email/Push уведомления** — для уведомлений используйте Resend, OneSignal и подобные через webhooks.

TopGun заменяет **слой данных**: хранение, синхронизацию, запросы и поиск. Всё остальное подключается через стандартные интеграции.

## Почему сейчас

Три тренда сходятся одновременно:

1. **Local-first набирает массу.** Ink & Switch popularized the concept, CRDT-библиотеки зреют (Yjs, Automerge), разработчики начинают требовать offline-first как стандарт, а не опцию. Новая волна стартапов (PowerSync, ElectricSQL, Instant, Triplit, Zero) подтверждает спрос — но ни один из них не достиг доминирования.

2. **Firebase fatigue.** Разработчики массово ищут альтернативы: vendor lock-in, непредсказуемый pricing, ограниченные запросы, отсутствие полнотекстового поиска. "Firebase alternative" — один из самых частых поисковых запросов в категории BaaS.

3. **Rust меняет серверную инфраструктуру.** Rust-серверы дают 10-100x лучшую производительность на доллар затрат по сравнению с Node.js/Elixir аналогами. Это позволяет соло-команде конкурировать по перформансу с VC-funded продуктами. Для контекста: Supabase Realtime написан на Elixir, ElectricSQL — на Elixir, PowerSync — на Dart/Kotlin.

## Технический фундамент

Сервер написан на Rust (axum + tokio). Клиент — TypeScript (React SDK, Node.js, браузер).

- **CRDT (LWW-Map, OR-Map)** с Hybrid Logical Clock — автоматическое разрешение конфликтов без центрального арбитра
- **Merkle tree sync** — при reconnect передаются только дельты, не полный dataset
- **DataFusion SQL** — полноценный SQL engine на сервере, доступный из SDK через `client.sql()` и из админ-панели
- **Tantivy FTS** — полнотекстовый поиск с BM25 scoring
- **Кластерная архитектура** — 271 партиция, горизонтальное масштабирование без смены кода приложения
- **500K+ ops/sec throughput, sub-2ms median write latency** на одном узле (200 WebSocket connections, in-process benchmark, M1 Max)
