# Research: Per-Partition Data Ownership Architecture

## Summary

TopGun's SPEC-116 (partition-based MPSC dispatch) correctly routes operations to per-partition workers, but all workers still contend on shared global data structures: `RecordStoreFactory` (DashMap), `MerkleSyncManager` (DashMap + per-tree Mutex), and Tantivy search indexes (`RwLock<HashMap<String, TantivyMapIndex>>`). The Tantivy `commit()` call (20-50ms WRITE lock per PUT) is the dominant bottleneck, reducing theoretical throughput from ~67,000 ops/sec to ~50 ops/sec. The recommended solution is a **PartitionWorkerContext** struct that bundles per-worker-owned data (RecordStore cache, MerkleTree instances, per-partition Tantivy indexes), eliminating cross-worker lock contention on the write path. Cross-partition queries use a scatter-gather pattern over the worker set.

## Background

### The Problem

k6 throughput tests show 102 ops/sec with 200 VUs — identical to pre-SPEC-116 numbers. The PartitionDispatcher successfully routes operations to N workers via bounded MPSC channels, and each worker owns its own `OperationPipeline` (Tower middleware stack). However, the domain services (`CrdtService`, `SyncService`, `SearchService`) are `Arc`-shared across all workers and access global shared data structures:

| Data Structure | Lock Type | Hold Time | Contention |
|---|---|---|---|
| `SearchMutationObserver::indexes` | `RwLock<HashMap>` WRITE | 20-50ms (tantivy commit) | CRITICAL |
| `MerkleSyncManager::lww_trees` | `DashMap` + `parking_lot::Mutex` per tree | 15us | HIGH (partition 0 funnel) |
| `RecordStoreFactory::store_cache` | `DashMap` | 2us | LOW |
| `HashMapStorage::entries` | `DashMap` | 1-2us | LOW |

The Tantivy commit on every PUT serializes all 10 workers through a single WRITE lock, making partition dispatch pointless. Even without search, the Merkle partition 0 dual-write creates a funnel where all partition workers contend on the same Mutex.

### What Hazelcast Does

Hazelcast solves this architecturally with a three-layer ownership model:

**Layer 1: PartitionOperationThread** (`spi/impl/operationexecutor/impl/PartitionOperationThread.java`)
- N threads, each with its own `OperationQueue`
- Operations are routed by `partitionId % threadCount`
- Each thread has an `OperationRunner[]` array indexed by partition ID

**Layer 2: PartitionContainer** (`map/impl/PartitionContainerImpl.java`)
- One per partition (271 in both Hazelcast default and TopGun)
- Owns a `ConcurrentMap<String, RecordStore>` — maps name to per-map store
- Created eagerly at startup: `partitionContainers[i] = new PartitionContainerImpl(service, i)`
- Since the partition thread is the only writer, the ConcurrentMap is effectively single-threaded

**Layer 3: Per-partition indexes** (`MapContainerImpl.java`)
- `shouldUseGlobalIndex()` determines whether indexes are per-partition or global
- Default: per-partition indexes via `partitionedIndexRegistry.computeIfAbsent(partitionId, ...)`
- Index is created inside `PartitionContainerImpl.createRecordStore()` — owned by the partition

**Key insight:** Hazelcast does NOT share data structures between partition threads. The PartitionContainer is the ownership boundary. Data flows IN via the operation queue and OUT via responses/broadcasts. This makes the write path completely lock-free within each partition.

## Options Explored

### Option 1: PartitionWorkerContext (Full Ownership)

**Description:** Create a `PartitionWorkerContext` struct that bundles all per-worker-owned data. Each worker task owns one context exclusively. The `pipeline_factory` closure in `PartitionDispatcher::new()` creates a context per worker. Domain services receive a `&PartitionWorkerContext` reference instead of accessing global shared structures.

```
struct PartitionWorkerContext {
    // Which partitions this worker is responsible for
    assigned_partitions: Vec<u32>,

    // Per-partition record stores (HashMap, not DashMap — single-threaded access)
    record_stores: HashMap<(String, u32), Arc<dyn RecordStore>>,

    // Per-partition Merkle trees (no Mutex — single-threaded access)
    lww_trees: HashMap<(String, u32), MerkleTree>,
    ormap_trees: HashMap<(String, u32), ORMapMerkleTree>,

    // Per-worker Tantivy indexes (one index per map, covers this worker's partitions)
    search_indexes: HashMap<String, TantivyMapIndex>,

    // Shared (read-only or atomic) references
    data_store: Arc<dyn MapDataStore>,
    connection_registry: Arc<ConnectionRegistry>,
}
```

**Architecture change:**
- `PartitionDispatcher::new()` takes a `context_factory: Fn(worker_id, assigned_partitions) -> PartitionWorkerContext`
- Each worker task owns `(OperationPipeline, PartitionWorkerContext)`
- Domain services become stateless — they receive context via `Operation` or via worker-local state
- `RecordStoreFactory`, `MerkleSyncManager` are replaced by worker-local `HashMap`s
- Cross-partition queries (QueryService, SearchService) use scatter-gather via MPSC channels

**Pros:**
- Zero lock contention on write path — matches Hazelcast architecture exactly
- Theoretical throughput matches hardware limits (~67,000 ops/sec for in-memory, no search tax)
- Tantivy commit only blocks the owning worker, not all workers
- MerkleTree access is lock-free within each worker
- Clean ownership model — Rust's type system enforces single-owner semantics

**Cons:**
- Large refactoring: domain services (`CrdtService`, `SyncService`, `SearchService`) need to accept context instead of Arc-shared state
- Cross-partition queries require a scatter-gather protocol (new complexity)
- Merkle partition 0 dual-write must be redesigned (see Trade-offs section)
- Memory overhead: each worker has its own Tantivy index set (mitigated: indexes are RAM-backed, data is already in memory)
- 271 partitions / N workers means each worker manages ~27 partitions (with 10 workers)
- `MutationObserver` trait is synchronous — cannot easily be called with borrowed context across workers

### Option 2: Sharded Locks (Incremental Fix)

**Description:** Keep the current architecture but eliminate the worst bottlenecks:
1. Replace global `RwLock<HashMap<String, TantivyMapIndex>>` with `DashMap<String, TantivyMapIndex>`
2. Batch Tantivy commits: accumulate writes and commit every N ms instead of per-PUT
3. Use `DashMap` sharding to reduce Merkle partition 0 contention

**Pros:**
- Minimal code changes — no architectural refactoring
- Tantivy batching alone could yield 10-50x improvement
- Lower risk — incremental changes, easy to test

**Cons:**
- Does not eliminate contention — only reduces hold time
- Tantivy `IndexWriter` is `!Sync` — only one writer per index, so DashMap does not help with write contention
- Merkle partition 0 remains a funnel (all partitions dual-write to partition 0)
- Does not scale with worker count — adding more workers increases contention
- Architectural debt: TopGun diverges further from Hazelcast's proven model

### Option 3: Async Tantivy via Dedicated Writer Task

**Description:** Decouple Tantivy indexing from the write path entirely. The `SearchMutationObserver` sends index events to a dedicated per-map writer task via an unbounded channel. The writer task batches documents and commits periodically.

**Pros:**
- Eliminates Tantivy commit from the PUT hot path entirely
- Searches become eventually consistent (acceptable for full-text search)
- Can be combined with Option 1 or Option 2
- Already partially implemented: `SearchMutationObserver` has a batch processor for notification delivery

**Cons:**
- Search results lag behind writes by the batch interval
- Existing batch processor handles notifications, not indexing — `index_document()` + `commit()` still happen synchronously in `on_put()`
- Does not address Merkle or RecordStore contention
- Single writer task per map could itself become a bottleneck under heavy write load

### Option 4: Hybrid — PartitionWorkerContext + Async Search

**Description:** Combine Option 1 (full per-worker data ownership) with Option 3 (async Tantivy indexing). The PartitionWorkerContext owns RecordStore and MerkleTree per-partition. Tantivy indexing is decoupled from the write path via a per-worker channel to a dedicated indexer task (one per worker, not one global).

**Pros:**
- Best of both worlds: zero-contention write path + eventual search indexing
- Tantivy commit amortized across many writes per batch
- Per-worker indexer means no cross-worker contention on Tantivy either
- Cross-partition search queries scatter across workers, each searching its local index

**Cons:**
- Most complex implementation
- Search results are eventually consistent
- Cross-partition query scatter-gather adds latency to search queries

## Codebase Findings

### Current Data Flow (PUT hot path)

1. WebSocket handler receives `ClientOp`, classifies via `OperationService`
2. `PartitionDispatcher::dispatch()` routes to worker by `partition_id % worker_count`
3. Worker's `OperationPipeline` (Tower stack) calls `CrdtService::handle_client_op()`
4. `CrdtService::apply_single_op()` calls `self.record_store_factory.get_or_create(map, partition)`
   - **LOCK 1:** `RecordStoreFactory::store_cache` DashMap (fast, ~2us)
5. `store.put()` calls `HashMapStorage::put()` then `CompositeMutationObserver::on_put()`
   - **LOCK 2:** `HashMapStorage::entries` DashMap (fast, ~1-2us)
   - Observer fan-out calls (synchronous, blocking the worker):
     - **LOCK 3:** `MerkleMutationObserver::update_tree()` -> `MerkleSyncManager::update_lww()` -> DashMap entry + Mutex lock (~15us per tree, x2 for dual-write)
     - **LOCK 4:** `SearchMutationObserver::index_and_notify()` -> `self.indexes.write()` (RwLock WRITE) -> `index.index_document()` + `index.commit()` (20-50ms total)
     - **LOCK 5:** `QueryMutationObserver` (sends events, relatively cheap)

### Key Files and Their Roles

- **`packages/server-rust/src/service/dispatch.rs`** — `PartitionDispatcher`: routes ops to workers. Currently takes `pipeline_factory: Fn() -> OperationPipeline`. Needs to take a `context_factory` for per-worker data.

- **`packages/server-rust/src/storage/factory.rs`** — `RecordStoreFactory`: global `DashMap<(String, u32), Arc<dyn RecordStore>>` cache. In Option 1, this becomes a per-worker `HashMap` inside `PartitionWorkerContext`.

- **`packages/server-rust/src/storage/merkle_sync.rs`** — `MerkleSyncManager`: global `DashMap<(String, u32), Mutex<MerkleTree>>`. In Option 1, trees become per-worker `HashMap` entries without Mutex.

- **`packages/server-rust/src/service/domain/search.rs`** — `SearchService` + `SearchMutationObserver`: the CRITICAL bottleneck. `indexes: Arc<RwLock<HashMap<String, TantivyMapIndex>>>` is shared across all workers. `on_put()` takes a WRITE lock and calls `commit()` synchronously.

- **`packages/server-rust/src/bin/test_server.rs`** — `build_services()`: shows the current wiring pattern. All domain services are `Arc`-wrapped and shared via `pipeline_factory` closure. The wiring needs to change to per-worker construction.

- **`packages/server-rust/src/storage/engines/hashmap.rs`** — `HashMapStorage`: uses `DashMap` internally. If owned by a single worker, could be replaced with `HashMap` for better cache locality and no atomic overhead.

### Hazelcast Reference Files

- **`hazelcast/map/impl/PartitionContainer.java`** — Interface defining per-partition data ownership. Key method: `getRecordStore(name)` returns a store from the partition's local map.

- **`hazelcast/map/impl/PartitionContainerImpl.java`** — Implementation: owns `ConcurrentMap<String, RecordStore> maps` per partition. `createRecordStore()` also creates per-partition indexes when `!shouldUseGlobalIndex()`.

- **`hazelcast/map/impl/MapContainerImpl.java`** — `partitionedIndexRegistry: ConcurrentMap<Integer, IndexRegistry>` — per-partition index registries. The dual model (global vs partitioned) is configurable.

- **`hazelcast/spi/impl/operationexecutor/impl/PartitionOperationThread.java`** — Thread that processes partition operations. Has `OperationRunner[]` array indexed by partition ID for routing within the thread.

- **`hazelcast/map/impl/MapServiceContextImpl.java`** — Creates `PartitionContainer[]` at startup (one per partition count). `getRecordStore(mapName, partitionId)` delegates to `getPartitionContainer(partitionId).getRecordStore(mapName)`.

## Trade-offs Analysis

| Aspect | Option 1: Full Ownership | Option 2: Sharded Locks | Option 3: Async Tantivy | Option 4: Hybrid |
|--------|--------------------------|------------------------|------------------------|------------------|
| Write throughput | ~67,000 ops/sec | ~5,000-10,000 ops/sec | ~67,000 ops/sec (without search) | ~67,000 ops/sec |
| Lock contention | Zero on write path | Reduced but not eliminated | Eliminated for search only | Zero on write path |
| Search consistency | Immediate | Immediate | Eventually consistent | Eventually consistent |
| Implementation effort | High (3-4 specs) | Low (1 spec) | Medium (1-2 specs) | High (3-4 specs) |
| Merkle p0 fix | Required (redesign dual-write) | Not addressed | Not addressed | Required (redesign dual-write) |
| Cross-partition queries | Scatter-gather needed | No change needed | No change for queries | Scatter-gather needed |
| Matches Hazelcast model | Yes | No | No | Yes |
| Memory overhead | Slightly higher (per-worker copies) | None | None | Slightly higher |
| Future scalability | Excellent (add workers = more throughput) | Poor (more workers = more contention) | Good for search | Excellent |

### Merkle Partition 0 Dual-Write Problem

Currently, every Merkle update dual-writes to both the record's actual partition AND partition 0 (the "client-sync aggregate"). This means ALL partition workers contend on partition 0's Mutex. In the per-worker-ownership model, this breaks because partition 0's tree is owned by whichever worker handles `0 % worker_count`.

**Solutions:**

1. **Per-worker aggregate tree:** Each worker maintains its own partition-0 equivalent for the partitions it owns. Client sync queries merge results from all workers. Adds scatter-gather complexity but eliminates the contention point entirely.

2. **Deferred partition-0 update:** Update only the partition-N tree synchronously. A background task periodically rebuilds partition 0 from all partition trees. Trades consistency for throughput.

3. **Eliminate partition 0 entirely:** Clients do delta sync against individual partitions (scatter across all 271, or only the changed ones). This is how Hazelcast works — there is no aggregate partition. The client maintains a per-partition Merkle hash set and diffs each partition independently.

**Recommendation for partition 0:** Option 3 (eliminate partition 0) is the cleanest. The TS client already supports per-partition sync via `MerkleTree`. The current partition 0 aggregate was a shortcut to avoid changing the client. Since this is a performance-critical refactoring, eliminating the shortcut is justified.

### Cross-Partition Query Scatter-Gather

When a query or search spans multiple partitions, the requesting service must:
1. Identify which workers own the relevant partitions
2. Send sub-queries to each worker via their MPSC channels
3. Collect and merge results

**Implementation approach:**
- Add a `PartitionQuery` operation variant that carries the query and a oneshot response channel
- Each worker evaluates the query against its local stores and sends results back
- The originating service merges results (sort, limit, deduplicate)
- This is exactly how Hazelcast's `QueryRunner` works: partition-level query + merge at coordinator

**Performance impact:** Adds one MPSC round-trip per worker (~1-5us). For a 10-worker setup querying all partitions, this adds ~50us latency to cross-partition queries — negligible compared to the current 20-50ms Tantivy lock.

## Recommendations

**Recommended approach:** Option 4 (Hybrid: PartitionWorkerContext + Async Tantivy)

**Reasoning:**

1. **Matches the proven model.** Hazelcast's PartitionContainer pattern has been battle-tested across millions of production deployments. TopGun already copied the dispatch routing but missed the data ownership — completing the model is the architecturally correct solution.

2. **Eliminates the dominant bottleneck.** Tantivy commit (20-50ms) accounts for >99% of the write-path latency. Async indexing removes it from the hot path entirely. Per-worker ownership removes the remaining DashMap/Mutex contention.

3. **Scales linearly with workers.** Adding more workers increases throughput proportionally because there is no shared state. This is the thread-per-core pattern described in the literature.

4. **Future-proof.** When DataFusion SQL and DAG execution arrive (v2.0), per-partition data ownership provides the natural shard boundary for distributed query execution.

5. **Eventually-consistent search is acceptable.** Full-text search is inherently best-effort. A 16ms batch delay is imperceptible to users and enables 1000x better write throughput.

**Implementation strategy (phased):**

**Phase A: Async Tantivy indexing (immediate win, 1 spec)**
- Move `index_document()` + `commit()` out of `SearchMutationObserver::on_put()` into the existing background batch processor
- Batch documents and commit every 16ms (already has `batch_interval_ms` parameter)
- This alone should yield ~10-50x write throughput improvement
- No architectural changes needed — pure observer refactoring

**Phase B: PartitionWorkerContext struct (2 specs)**
- Define `PartitionWorkerContext` trait and concrete struct
- Move `RecordStoreFactory::store_cache` into per-worker context as `HashMap`
- Move `MerkleSyncManager` trees into per-worker context as plain `HashMap` (no Mutex)
- Refactor `PartitionDispatcher::new()` to accept context factory
- Domain services accept `&PartitionWorkerContext` or extract from `Operation`

**Phase C: Eliminate Merkle partition 0 (1 spec)**
- Remove dual-write from `MerkleMutationObserver`
- Update `SyncService` to scatter delta-sync queries across workers
- Update TS client sync protocol if needed (may already support per-partition sync)

**Phase D: Cross-partition query scatter-gather (1 spec)**
- Add `PartitionQuery` operation variant
- Implement scatter-gather in `QueryService` and `SearchService`
- Merge results at coordinator (sort, limit, deduplicate)

**Implementation notes:**

- Phase A can ship independently and delivers the biggest performance win
- Phases B-D are the architectural refactoring and should be planned together
- The `MutationObserver` trait is `Send + Sync` but methods take `&self` — per-worker ownership means observers are created per-worker and never shared, so `Sync` becomes vacuous
- `HashMapStorage` can be simplified to `HashMap<String, Record>` (no `DashMap` needed) when owned by a single worker, improving cache locality
- The `ObserverFactory` pattern already supports per-(map, partition) observer creation — extend it to per-worker creation by adding a `worker_id` parameter
