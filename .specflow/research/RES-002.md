# Research: Hazelcast Merkle/AntiEntropy Per-Partition Ownership Model for TopGun Sync

## Summary

Hazelcast's Merkle trees are strictly per-(IMap, partition) with no global aggregate. WAN anti-entropy works by each partition's primary replica comparing its own tree against the remote independently — there is no partition 0 collecting all keys. TopGun can eliminate the partition 0 dual-write bottleneck by replacing it with a server-side scatter-gather at sync time: the client sends one `SYNC_INIT`, the server fans out to all partitions, merges the hashes into a synthesized root response, and subsequent bucket/leaf traversal is routed to the correct partition-specific tree. This requires no client protocol change and eliminates the hot-path Mutex.

## Background

The current TopGun dual-write design was introduced in SPEC-080 to fix Merkle partition mismatch (TODO-111). Every PUT is written to the key's actual partition tree AND to partition 0's tree. Partition 0 acts as a global aggregate so that client `SYNC_INIT` (which carries no partition hint) gets a single root hash covering all data.

The performance problem: all 271 partition workers compete for the same `Mutex<MerkleTree>` on partition 0. Under load this becomes a global serialization point. Benchmarks show throughput collapses from 196 ops/sec (1 VU) to 126 ops/sec (200 VUs) with p50=5.7s latency — partition worker parallelism (SPEC-116/118) is negated.

The research question: how should TopGun adopt a true per-partition Merkle model while keeping the partition-unaware TS client unchanged?

## Options Explored

### Option 1: Scatter-Gather Root Hash (Server synthesizes at sync time)

**Description:**
On `SYNC_INIT`, the server fans out to all N active partition trees for the requested `map_name`, XORs (or sums) all root hashes into a single synthetic root hash, returns that to the client. Bucket/leaf traversal messages carry an opaque `path` prefix that encodes the partition ID internally (e.g., `"042/abc"`). The server decodes the partition ID from the path prefix and routes to the correct per-partition tree.

**How this matches Hazelcast:** In Hazelcast WAN, the initiating cluster queries all partitions independently in parallel (each partition primary sends an anti-entropy operation independently). The receiver processes each partition independently. There is no global aggregator. TopGun can simulate this by having the SyncService act as the aggregator at query time — composing partition trees on-demand rather than maintaining a pre-computed aggregate.

**Pros:**
- Eliminates partition 0 Mutex entirely — no dual-write
- Per-partition trees are independently locked, enabling true parallelism
- Client protocol unchanged (SYNC_INIT still has only `map_name`)
- No data duplication in MerkleSyncManager
- Matches Hazelcast model faithfully

**Cons:**
- `SYNC_INIT` now requires O(N_partitions) hash reads to construct root response — but these are 271 non-blocking DashMap lookups on independent Mutexes, so they parallelize cleanly
- Synthetic root hash must be deterministic and stable between calls with the same data; XOR or wrapping-add of all partition root hashes works (same commutative property as Hazelcast's `sumHash`)
- Path encoding changes: `path` goes from a pure hex string like `"abc"` to `"042/abc"` or similar — but this is an internal server detail since the client echo's the path back opaquely

**Implementation complexity:** Medium. Changes to: `MerkleSyncManager.aggregate_root_hash()`, `SyncService::handle_sync_init()`, `SyncService::handle_merkle_req_bucket()`, `MerkleMutationObserver::update_tree()`. `DefaultRecordStore` is not touched.

### Option 2: Remove Dual-Write, Add Partition Hint to SYNC_INIT

**Description:**
Remove partition 0 entirely and add a `partitionId` field to `SYNC_INIT`. Client learns its partition assignment from a one-time handshake or uses a hash of `mapName` to pick a partition. Client sync covers only the client's assigned partition.

**Pros:**
- Simplest possible server change: just remove the dual-write
- Each client only syncs one partition tree — lower server fan-out per reconnect

**Cons:**
- Breaks client invariant: clients currently sync all data. A client that assigns to partition 5 would never see data written by a client assigned to partition 17.
- Client must become partition-aware, violating the product constraint
- Partial coverage means reconnecting clients miss data they should have

**Assessment:** Incompatible with current product design. Rejected.

### Option 3: Lazy Aggregate — Rebuild Partition 0 on Demand

**Description:**
Stop dual-writing to partition 0. On `SYNC_INIT`, build a temporary aggregate root hash from all partition trees (same scan as Option 1), but cache it in a short-lived Arc<AtomicU32> that's invalidated on any mutation (via an epoch counter in MerkleSyncManager).

**Pros:**
- Eliminates hot-path Mutex on partition 0 during writes
- Cached hash amortizes cost when multiple clients reconnect simultaneously

**Cons:**
- Epoch invalidation requires a global atomic increment on every PUT — a different but still shared contention point
- More complex than Option 1 without clear benefit: if the cache hit rate is low (clients reconnecting at different times), it degrades to Option 1 overhead anyway
- Added state management complexity

**Assessment:** Premature optimization. Option 1 is simpler and sufficient.

### Option 4: Client-Side Partition Routing (Full Hazelcast Client Model)

**Description:**
Give the client a partition table. Client knows that keys in bucket `[0..N)` go to partition ID X. Client sends separate `SYNC_INIT` per partition it owns data for.

**Pros:**
- True Hazelcast client model
- Minimizes server computation per sync
- Natural foundation for v2.0 partial replication / Shapes feature

**Cons:**
- Breaking client protocol change
- Increases reconnect latency: one round-trip per partition with data (up to 271 round-trips for full data set)
- Requires distributing and maintaining partition table on client side
- Belongs in v2.0 (Shapes/SchemaProvider), not now

**Assessment:** Correct long-term direction but out of scope for this fix. Revisit in v2.0 Shapes feature (SPEC-SchemaProvider).

## Codebase Findings

### TopGun dual-write location
`packages/server-rust/src/storage/merkle_sync.rs` — `MerkleMutationObserver::update_tree()` (lines 236–267). Dual-write to `Self::CLIENT_SYNC_PARTITION = 0` is the hot-path bottleneck. The constant is defined at line 229.

### SyncService partition_id usage
`packages/server-rust/src/service/domain/sync.rs` — `handle_sync_init()` (line 121) reads `ctx.partition_id.unwrap_or(0)`. Client sends no partition_id, so this always resolves to 0. Same pattern in `handle_ormap_sync_init()` (line 244) and `handle_merkle_req_bucket()` (line 150).

### DefaultRecordStore — clean
`packages/server-rust/src/storage/impls/default_record_store.rs` — does NOT dual-write. The observer pattern fires `self.observer.on_put()` which delegates to `MerkleMutationObserver`. The fix is purely in the observer, not in `DefaultRecordStore`.

### MerkleSyncManager structure
`packages/server-rust/src/storage/merkle_sync.rs` — `DashMap<(String, u32), Mutex<MerkleTree>>`. Key is `(map_name, partition_id)`. Each partition tree has its own `Mutex`. The DashMap shards are independent. Accessing partition 0 while another shard is locked is perfectly safe — the problem is that ALL workers converge on the same `(map_name, 0)` key.

### Client sync protocol
`packages/client/src/sync/MerkleSyncHandler.ts` — `sendSyncInit()` sends `{ type: 'SYNC_INIT', mapName, lastSyncTimestamp }`. No partition ID. The client echoes back the `path` string it receives from the server in `MERKLE_REQ_BUCKET`. The path is opaque to the client.

### Hazelcast Merkle architecture
`/Users/koristuvac/Projects/hazelcast/hazelcast/src/main/java/com/hazelcast/wan/impl/merkletree/ArrayMerkleTree.java` — one tree per (IMap, partition). NOT thread-safe by design — Hazelcast wraps each per-partition record store's tree inside the partition operation thread (single-owner pattern). Trees are updated synchronously on put/replace/remove.

`/Users/koristuvac/Projects/hazelcast/hazelcast/src/main/java/com/hazelcast/internal/partition/impl/PartitionPrimaryReplicaAntiEntropyTask.java` — anti-entropy is invoked with `partitionId` in the constructor and calls `retainAndGetNamespaces()` scoped to that partition. No cross-partition aggregation. Each partition is responsible for its own tree reconciliation.

`/Users/koristuvac/Projects/hazelcast/hazelcast/src/main/java/com/hazelcast/map/impl/operation/MapReplicationStateHolder.java` — differential replication uses `merkleTreeDiffByMapName` (node order/hash pairs per IMap). All comparisons are between two per-partition trees with the same partition scope.

`/Users/koristuvac/Projects/hazelcast/hazelcast/src/main/java/com/hazelcast/wan/impl/merkletree/MerkleTreeUtil.java` — hash combination is `addHash(original, added) = original + HUGE_PRIME * added` (commutative polynomial). `removeHash` is the inverse. `sumHash(left, right) = left + right`. All operations are order-independent, enabling correct multi-partition hash aggregation.

## Recommendations

**Recommended approach:** Option 1 — Scatter-Gather Root Hash

**Reasoning:**
1. Eliminates the partition 0 Mutex bottleneck without any client changes. The client sends `SYNC_INIT { mapName }`, receives `SYNC_RESP_ROOT { rootHash }`, and proceeds with the existing bucket traversal protocol — all unchanged.
2. Matches Hazelcast's actual architecture: Hazelcast has no global Merkle aggregate. Anti-entropy happens per-partition, and the initiator collects results from all partitions independently.
3. The synthetic root hash is trivially computed: iterate all `(map_name, *)` entries in `MerkleSyncManager.lww_trees`, collect root hashes, sum them (wrapping addition is commutative and matches the per-node hash combination used by `MerkleTree` internally).
4. The opaque `path` string the client echoes back can embed a partition prefix. Since the client never parses `path` (it just echoes it), adding `"042/"` prefix to the path is a pure server-internal convention.
5. The `DefaultRecordStore` is clean — no changes required there. Changes are confined to `MerkleMutationObserver`, `MerkleSyncManager`, and `SyncService`.

**Implementation notes:**

1. **Remove dual-write from `MerkleMutationObserver::update_tree()`** — delete the `if self.partition_id != Self::CLIENT_SYNC_PARTITION` blocks. Each observer writes only to `self.partition_id`. The `CLIENT_SYNC_PARTITION` constant and the partition 0 special-case logic are deleted entirely.

2. **Add `aggregate_root_hash()` to `MerkleSyncManager`** — iterates all entries keyed by `(map_name, *)`, reads each root hash (one `Mutex::lock()` per partition, each released immediately), returns `wrapping_add` of all hashes. Same for OR-Map. Since DashMap shards are independent, this is highly parallel.

3. **Update `SyncService::handle_sync_init()`** — call `merkle_manager.aggregate_root_hash(&map_name)` instead of `with_lww_tree(&map_name, 0, ...)`.

4. **Update `SyncService::handle_merkle_req_bucket()`** — decode partition ID from path prefix (`"042/abc"` → partition `42`, sub-path `"abc"`). For the root bucket request (path `""`), return aggregate of all partitions' bucket hashes (fan-out across partitions for the first level). For all subsequent traversals, the path carries the partition prefix so they route directly to one partition tree.

5. **Path encoding convention** — The root path `""` triggers aggregate mode. Any path starting with a 3-digit partition prefix (e.g. `"042"`) routes to that partition. The server adds the prefix when returning `SYNC_RESP_BUCKETS`; the client echoes it back unchanged. No client changes needed.

6. **Tests** — Remove the `dual_write_*` tests in `merkle_sync.rs` (they test the now-deleted behavior). Add `aggregate_root_hash_spans_all_partitions` test and `path_prefix_routing` test in `SyncService`.

7. **Migration safety** — Since `ctx.partition_id` from client messages is always `None` (clients don't send partition IDs), removing the `unwrap_or(0)` fallback is safe. Replace with the aggregate call.
