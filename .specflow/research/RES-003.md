# Research: Backpressure and Bounded Channel Patterns for High-Throughput Operation Dispatch

## Summary

The current dispatch architecture has bounded channels (1024 per worker) but no enforcement that prevents WebSocket handlers from enqueuing faster than workers can drain them. The core issue is that `sender.send().await` in `dispatch.rs` blocks the WebSocket inbound task until space is available in the worker's channel — but with 200 VUs each sending 20 batches/sec, over 4000 messages/sec compete for 10 workers, causing multi-second queue accumulation. The recommended fix is a three-layer approach: (1) replace `.send().await` with `try_send()` + `OperationError::Overloaded` at the dispatch boundary, (2) add a global in-flight semaphore capping concurrent dispatches per connection, and (3) return `OP_ACK` with an explicit overload error code so clients can back off.

## Background

After SPEC-119 eliminated Merkle partition 0 dual-write, throughput improved from 126 to 323 ops/sec at 200 VUs. However:

- k6 sends ~17,726 ops/sec, server ACKs only 323 ops/sec (1.8% throughput)
- p50 latency = 7.7 seconds — this is queue accumulation, not processing latency
- 200 VUs × 20 batches/sec × 10 ops/batch = 40,000 items/sec queued against workers that can process ~3,230/sec
- Each WebSocket inbound task calls `dispatch_message().await` sequentially, which calls `dispatcher.dispatch(op).await`, which calls `sender.send(request).await` — this blocks until the worker channel has space
- When all 10 worker channels are full (10 × 1024 = 10,240 items buffered), every inbound task blocks, causing TCP receive buffers to fill, triggering OS-level TCP backpressure — but only after enormous queue accumulation

The current `LoadShedLayer` (semaphore, default 1000 permits) operates inside the worker task. It doesn't shed load at the dispatch entry point; the operation must already be enqueued before the semaphore is checked. This means it protects against CPU overload inside a single worker, not against channel saturation across all workers.

## Hazelcast BackpressureRegulator Analysis

Source: `/Users/koristuvac/Projects/hazelcast/hazelcast/src/main/java/com/hazelcast/spi/impl/operationservice/impl/BackpressureRegulator.java`

Hazelcast uses a `CallIdSequence` with a global in-flight counter to gate new invocations:

1. **Global max concurrent invocations** = `(partitionCount + 1) × maxInvocationsPerPartition`
   (default: `(271 + 1) × 100 = 27,200` total in-flight)

2. **Backoff on exhaustion**: `CallIdSequenceWithBackpressure.handleNoSpaceLeft()` spins with exponential backoff up to 500ms, then throws `HazelcastOverloadException`. This is a **block-with-timeout** strategy: the caller waits briefly, then gets an error.

3. **Periodic sync forcing**: async backups are periodically converted to sync, forcing callers to wait for queues to drain. This is the primary mechanism that prevents accumulation — not rejection.

4. **Key insight**: Hazelcast blocks the invoking thread (or async task) for up to `backoffTimeoutMs` before rejecting. This is viable in Hazelcast's thread-per-connection model but problematic in a single-threaded async model like Tokio, where blocking tasks starves other futures.

5. **Key architectural difference**: Hazelcast calls are typically synchronous from the caller's perspective (client blocks waiting for response). TopGun WebSocket clients are event-driven and pipelined — they send batches without waiting for ACKs, making them far more aggressive producers.

## Arroyo BatchSender Analysis (Row-Bounded Channels)

Source: `/Users/koristuvac/Projects/rust/arroyo/crates/arroyo-operator/src/context.rs`

Arroyo uses a custom `BatchSender` wrapper around an `UnboundedSender` that enforces backpressure by counting rows rather than messages:

```rust
pub struct BatchSender {
    size: u32,                          // max queued rows
    tx: UnboundedSender<QueueItem>,     // underlying channel (unbounded)
    queued_messages: Arc<AtomicU32>,    // current count (rows)
    notify: Arc<Notify>,                // wakes blocked senders
}

// send() blocks until row count drops below limit
pub async fn send(&self, item: QueueItem) -> Result<(), SendError<QueueItem>> {
    loop {
        if cur + count <= self.size {
            // CAS to reserve space, then send
        } else {
            let notified = self.notify.notified();  // register wakeup FIRST
            // re-check (avoid lost-wakeup race)
            if space_available { continue; }
            notified.await;  // block until receiver dequeues
        }
    }
}
```

**Key lessons from Arroyo:**
- Row-based bounding is semantically richer than message-based bounding (one large batch ≠ one small batch)
- The lost-wakeup prevention pattern (register notify before re-checking) is the correct approach for async backpressure
- Arroyo uses this for **internal DAG edges** where blocking is safe; for external (client-facing) boundaries it is more aggressive

## TiKV ConcurrencyLimiter Analysis

Source: `/Users/koristuvac/Projects/rust/tikv/src/coprocessor/interceptors/concurrency_limiter.rs`

TiKV uses a semaphore-based approach with a grace period:
- Tasks are allowed to run for `time_limit_without_permit` (e.g., 100ms) before needing a permit
- After the grace period, the task suspends until a permit is available
- This allows light tasks (short ops) to complete without queuing, while heavy tasks are throttled

**Key lesson**: The grace period idea is useful for mixed workloads. For TopGun where all ops are roughly equal cost, a simpler approach suffices.

## Options Explored

### Option 1: try_send + immediate reject at dispatch boundary (Fail-Fast)

**Description:** Replace `sender.send(request).await` in `PartitionDispatcher::dispatch()` with `sender.try_send(request)`. If the worker channel is full, return `OperationError::Overloaded` immediately without blocking. The WebSocket handler translates this to an `OP_ERROR` with code 429 (or a new `OP_OVERLOADED` message type), and the client is responsible for retrying with exponential backoff.

**Implementation change in dispatch.rs:**
```rust
// Before:
sender.send(request).await.map_err(|_| OperationError::Internal(...))?;

// After:
sender.try_send(request).map_err(|e| match e {
    mpsc::error::TrySendError::Full(_) => OperationError::Overloaded,
    mpsc::error::TrySendError::Closed(_) => OperationError::Internal(...),
})?;
```

**Pros:**
- Zero latency to the caller — no blocking, no queue accumulation
- Inbound WebSocket task always makes progress (reads next frame immediately)
- Natural feedback: clients measure ack rate, back off when seeing 429 errors
- CRDT ordering preserved: rejected ops are retried from client's local log with original HLC timestamps
- Works with existing LoadShedLayer (defense in depth)

**Cons:**
- Requires client-side retry logic with backoff (client currently has no backpressure handling)
- Rejected ops increase network traffic (error ACK + retry)
- A poorly behaved client that ignores rejections will keep hammering the server
- Under moderate load (queue partially full), random rejection even when workers would have caught up

### Option 2: Per-connection in-flight semaphore (Hazelcast-style)

**Description:** Add a `tokio::sync::Semaphore` per connection (or globally across all connections) limiting how many operations can be in-flight simultaneously. Before calling `dispatcher.dispatch()`, the WebSocket handler tries to acquire a permit. If none is available, it either waits (blocking the inbound task) or rejects.

**Pros:**
- Naturally limits total concurrent operations server-wide
- Familiar pattern — already used by `LoadShedLayer`
- Can be tuned per-connection (fair allocation)

**Cons:**
- Global semaphore creates a new contention point with 200+ connections competing
- Per-connection semaphore requires state on `ConnectionHandle`, increasing complexity
- If semaphore acquisition blocks (awaiting), we reintroduce the queue-accumulation problem at a different layer
- Doesn't account for operation cost variation (one OpBatch with 10 ops vs. one sync query)

### Option 3: Connection-level rate limiting with token bucket (Tower rate-limit layer)

**Description:** Add a `tower::limit::RateLimitLayer` per connection, or a global rate limiter, that caps operations/sec server-wide. Excess operations are queued in Tower's internal queue or shed.

**Pros:**
- Smooth throughput ceiling
- Tower already in use in the pipeline

**Cons:**
- Tower's rate limiter queues excess operations internally — trades one unbounded queue for another
- Rate limiting is per-time-period, not per-capacity — doesn't directly solve queue accumulation
- Hard to set the right limit without knowing actual worker throughput

### Option 4: Bounded channel + connection admission control (Recommended combination)

**Description:** Combine Option 1 (try_send at dispatch boundary) with a global `Arc<Semaphore>` in `PartitionDispatcher` that limits total in-flight dispatches across all workers. The semaphore has `worker_count × channel_buffer_size` permits. `dispatch()` tries to acquire a permit with `try_acquire()` (non-blocking); if unavailable, returns `Overloaded`. This prevents the case where one hot partition's channel is full while others are empty.

**Pros:**
- Combines channel-level (per-worker) and system-level (global) limits
- Non-blocking — preserves inbound task throughput
- Single clear rejection point (in `PartitionDispatcher::dispatch()`)
- Global semaphore provides aggregate backpressure across uneven partition load

**Cons:**
- Two limits to tune (per-channel + global)
- Global semaphore can still be a contention point, though `try_acquire()` is lock-free on modern tokio

### Option 5: Reduce channel_buffer_size dramatically (32–64) so LoadShed kicks in sooner

**Description:** Change `channel_buffer_size` from 1024 to 32 or 64 per worker. With smaller buffers, workers reach capacity sooner. The current blocking `send().await` would then block inbound tasks sooner, triggering TCP backpressure from the OS earlier — at a lower total queue depth.

**Pros:**
- Minimal code change
- Uses existing TCP backpressure mechanism
- Queue depth shrinks from 10,240 items to 320–640 items

**Cons:**
- Still fundamentally a blocking approach — inbound tasks stall when channel is full
- Does not prevent queue accumulation, just reduces its maximum depth
- Under burst load, clients still see high latency (just lower max latency)
- 200 concurrent connections × blocking = 200 tasks contending on 10 tiny channels

## Trade-offs Analysis

| Aspect | Option 1 (try_send) | Option 2 (semaphore) | Option 3 (rate limit) | Option 4 (try_send + global semaphore) | Option 5 (smaller buffer) |
|--------|--------------------|-----------------------|-----------------------|----------------------------------------|---------------------------|
| Queue accumulation | Eliminated | Reduced | Not solved | Eliminated | Reduced |
| Inbound task blocking | None | Possible (await) | Possible (queue) | None | Yes |
| Code complexity | Low | Medium | Low | Medium | Trivial |
| Client retry required | Yes | No | No | Yes | No |
| CRDT ordering preserved | Yes (HLC retry) | Yes | Yes | Yes | Yes |
| Throughput under load | Max server capacity | Capped by semaphore | Capped by rate | Max server capacity | Lower (blocking) |
| Latency under load | Low (fast reject) | Variable | Variable | Low (fast reject) | High (blocking) |
| Fair per-connection | No (first-come) | Yes (per-conn) | Configurable | No (global) | No (first-come) |

## Recommendations

**Recommended approach:** Option 1 (try_send + reject) as primary, with Option 5 (reduce buffer to 256) as a secondary tuning to limit maximum queue depth if try_send is not implemented first.

**For the spec, implement:**

1. **Change `PartitionDispatcher::dispatch()` to use `try_send()`** — returns `OperationError::Overloaded` immediately if worker channel is full. No new types needed; `OperationError::Overloaded` already exists and is used by `LoadShedLayer`.

2. **In `dispatch_message()` (websocket.rs), map `OperationError::Overloaded` to an `OP_ERROR` response with HTTP code 429** so the client knows to back off. The error propagates through the existing `dispatch error` path and can use `ErrorPayload { code: 429, message: "server overloaded" }`.

3. **Reduce `DispatchConfig::channel_buffer_size` from 1024 to 256** — this is not a correctness change but limits worst-case queue depth (10 workers × 256 = 2,560 items max, ~800ms at 3,230 ops/sec versus 7.7 seconds currently).

4. **Expose `channel_buffer_size` and `worker_count` in the admin API** alongside the existing `max_concurrent_operations` so operators can tune without redeployment.

**Do NOT implement:**
- Blocking per-connection semaphores — they replicate the queue problem at a higher level
- Global rate limiting — does not address the structural mismatch between producer rate and consumer rate
- Arroyo-style row-bounded channels — over-engineering for TopGun's uniform op cost model

**Reasoning:**
- `OperationError::Overloaded` already exists, `LoadShedLayer` already uses it, and `dispatch_message()` already handles errors — minimal blast radius
- `try_send()` is a one-line change in `PartitionDispatcher::dispatch()` that eliminates queue accumulation structurally
- CRDT clients already have retry logic in `SyncEngine` (offline queue, reconnect sync) — they can absorb transient rejections without data loss
- The HLC timestamp is set at classification time (before dispatch), so a retried operation preserves causal ordering
- The existing LoadShedLayer remains as a second defense layer for per-worker overload independent of channel state

**Implementation notes:**
- `PartitionDispatcher::dispatch()` — swap `sender.send(request).await` for `sender.try_send(request)` and map `TrySendError::Full` → `OperationError::Overloaded`
- `dispatch_message()` in websocket.rs — the existing `Err(e)` branch already logs and returns; update to serialize an `ErrorPayload { code: 429 }` outbound message for `Overloaded` specifically
- `dispatch_op_batch()` — one sub-batch failing with `Overloaded` should fail the whole batch with a single 429 error (not partially apply)
- `DispatchConfig` — reduce default `channel_buffer_size` to 256 and document the tuning guidance
- New test: verify that a full channel returns `OperationError::Overloaded` synchronously without blocking
