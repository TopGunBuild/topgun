# Research: Shape Subscription Limits and Type-Safe Filters

## Summary

No major offline-first sync project enforces a hard default limit on shape/subscription queries. ElectricSQL, PowerSync, and Replicache all trust the developer to scope data appropriately, relying on filters and server-side sync rules rather than row caps. For type safety, the industry is split: ElectricSQL uses loose string-based SQL filters, PowerSync delegates to Kysely/Drizzle, and Prisma/Drizzle represent the gold standard with codegen-derived typed filter objects. TopGun should adopt a server-side configurable max limit (default 10,000) with a Zod-schema-derived generic type approach for client-side type safety.

## Background

TopGun shapes allow clients to subscribe to filtered subsets of map data (partial replication). Two open design questions:

1. **Limits:** Should there be a default/max limit on records returned in a shape response? The current implementation supports an optional `limit: Option<u32>` in `SyncShape` but has no server-enforced default or maximum.
2. **Type safety:** The current `ShapeSubscribeOptions.filter` is typed as `PredicateNode` (a recursive expression tree with string attribute names and `rmpv::Value` values). There is no generic type parameter tying the filter to a schema.

## Part 1: Limits and Pagination

### Project-by-Project Findings

#### ElectricSQL

- **Default limit:** None. All matching rows are returned when no `limit` is specified.
- **Server-side max:** No documented hard cap on shape size.
- **Pagination:** Available only via `requestSnapshot()` on `changes_only` streams (not on the main shape subscription). Supports `limit`, `offset`, and `orderBy` parameters.
- **Backpressure:** Shape logs are divided into fixed-size chunks. Large parameter sets can use POST to avoid 414 errors. No explicit rejection of "too large" shapes.
- **Philosophy:** Shapes are the unit of sync. Developer is responsible for scoping shapes to reasonable sizes via `where` clauses. Electric focuses on throughput optimization rather than hard limits.

#### PowerSync

- **Default limit:** No per-query row limit. The constraint is on buckets: 1,000 buckets per user/client.
- **Server-side max:** No documented per-bucket row limit. Sync Rules (legacy) and Sync Streams (beta) both rely on SQL-based data filtering.
- **TTL:** Each subscription has a configurable TTL (default 24 hours) that keeps data cached after unsubscribe, acting as a warm cache.
- **Backpressure:** Bucket-count limit is the primary protection. Each unique (stream, parameters) combination creates one bucket.
- **Philosophy:** Sync Rules declaratively define what data belongs to which user. The developer controls data volume through rule design.

#### Replicache

- **Default limit:** None. The entire Client View is synced via pull endpoint.
- **Server-side max:** Browser-limited. Supports hundreds of MB in practice (real-world: ~45MB for 18 months of team data).
- **Pagination:** Pull endpoint returns diffs (patches), not full snapshots after initial sync. No built-in limit parameter.
- **Backpressure:** None at the protocol level. Server's pull endpoint controls what goes into the Client View.
- **Philosophy:** "Bring Your Own Backend" - the server controls what data is in the Client View. No client-side limit mechanism.

#### Firebase Firestore

- **Default limit:** None. Without `.limit()`, all matching documents are returned.
- **Server-side max:** Previously had 100,000 results per query limit (since removed). Individual documents capped at 1MB. Collection depth limit applies.
- **Pagination:** `limitToFirst(n)`, `limitToLast(n)`, cursor-based pagination via `startAt`/`endAt`.
- **Backpressure:** Per-document size limit (1MB). Billing acts as natural brake (reads cost money).
- **Philosophy:** Pay-per-read model naturally limits over-fetching. SDK provides explicit limit/pagination methods.

#### Firebase Realtime Database

- **Default limit:** None. All matching data at a path is returned.
- **Server-side max:** 75 million nodes per listener path. Individual writes limited to 16MB.
- **Pagination:** `limitToFirst(n)`, `limitToLast(n)` methods.
- **Backpressure:** 75M node cap per path is a hard server limit. Bandwidth limits apply.

#### Hazelcast Near Cache

- **Default limit:** `max-size` defaults to `Integer.MAX_VALUE` (effectively unlimited) for on-heap; `10,000` for NATIVE in-memory format.
- **Eviction policies:** LRU (default), LFU, RANDOM, NONE.
- **Max-size policies:** ENTRY_COUNT (default), USED_NATIVE_MEMORY_SIZE, USED_NATIVE_MEMORY_PERCENTAGE, FREE_NATIVE_MEMORY_SIZE, FREE_NATIVE_MEMORY_PERCENTAGE.
- **Philosophy:** Near Cache is a client-side optimization. Server has separate eviction for IMap. Config-driven, not per-query.

#### LiveStore

- **Default limit:** Not applicable in the same way. Uses reactive SQLite queries locally. All data is in local SQLite.
- **Sync mechanism:** Event-sourcing with push/pull log sync. No shape/subscription concept - you query local SQLite directly.
- **Philosophy:** Full local replica via event log. Query locally with SQL.

### Limits Comparison Table

| Project | Default Limit | Server Max | Pagination | Backpressure Mechanism |
|---------|--------------|------------|------------|----------------------|
| ElectricSQL | None | None documented | `requestSnapshot()` only | Chunked shape logs |
| PowerSync | None (1K bucket limit) | None per bucket | N/A (sync rules filter) | Bucket count limit |
| Replicache | None | Browser-limited (~100s MB) | Pull diffs | Server controls Client View |
| Firestore | None | ~1MB/doc | `limit()`, cursors | Billing, doc size cap |
| Firebase RTDB | None | 75M nodes/path | `limitToFirst/Last` | Path node cap |
| Hazelcast Near Cache | MAX_INT / 10K (native) | Configurable | N/A (cache, not query) | Eviction policies |
| LiveStore | N/A | N/A | SQL LIMIT/OFFSET | N/A (local SQLite) |

## Part 2: Type-Safe Filters and Field Projections

### Project-by-Project Findings

#### ElectricSQL

- **Filter typing:** String-based SQL WHERE clauses. `where?: string` in TypeScript. No compile-time validation of column names or types.
- **Column projection:** `columns?: string[]`. No compile-time validation.
- **Generic type:** `ShapeStream<T>` accepts a row type generic, but it only types the *output* data, not the filter inputs.
- **Assessment:** Loose typing. Developer must know column names and SQL syntax. No schema-derived type safety.

#### PowerSync

- **Filter typing:** Delegates to ORM integrations (Kysely or Drizzle). Sync Rules themselves are YAML with SQL snippets (string-based).
- **Column projection:** Via ORM query builder or raw SQL.
- **Generic type:** Schema defined in TypeScript maps columns to types. Kysely provides `db.selectFrom('users').where('status', '=', 'active')` with autocomplete.
- **Assessment:** Type safety through ORM layer, not built into sync protocol. Schema definition is separate from query types.

#### Replicache

- **Filter typing:** No built-in query/filter language. Subscriptions are arbitrary functions: `subscribe(tx => tx.scan({prefix: 'todo/'}).toArray())`.
- **Column projection:** Not applicable. Subscriptions return whatever the function returns.
- **Generic type:** `Replicache<MD>` where `MD` is the mutator definitions. Subscriptions are untyped functions over the key-value store.
- **Assessment:** Maximum flexibility, minimal type safety for reads. Type safety is on mutations only.

#### Prisma (reference: query typing gold standard)

- **Filter typing:** Codegen-derived `UserWhereInput` types. Each field gets its own filter operators: `{ email: { contains: "gmail" }, age: { gte: 18 } }`. Full compile-time validation.
- **Column projection:** `select: { id: true, name: true }` returns `Pick<User, 'id' | 'name'>`. Return type narrows automatically.
- **Generic type:** `prisma.user.findMany({ where, select })` - types flow from schema through codegen. Not generic parameters but generated method signatures.
- **Assessment:** Best-in-class type safety. Requires codegen step. Complex TypeScript gymnastics internally.

#### Drizzle ORM (reference: query typing)

- **Filter typing:** Schema-derived. `eq(users.email, 'test@test.com')` uses column references. TypeScript validates column types.
- **Column projection:** `db.select({ id: users.id, name: users.name })` returns typed subset. Also supports `getColumns()` helper for spread/exclude patterns.
- **Generic type:** Types derived from schema definition. `const users = pgTable('users', { id: serial('id'), name: text('name') })` generates types.
- **Assessment:** Strong type safety without codegen. Schema-as-code approach. Slightly more verbose than Prisma but no build step.

#### tRPC (reference: inference patterns)

- **Filter typing:** Input validated via Zod schemas. Type inference flows from server to client.
- **Limitations:** Cannot define generic router procedures that accept different types per call. Static analysis at router definition time.
- **Assessment:** Excellent for RPC patterns. Relevant to TopGun because TopGun already uses Zod for schema validation.

### Type Safety Comparison Table

| Project | Filter Typing | Field Projection | Schema Source | Codegen Required |
|---------|--------------|-----------------|--------------|-----------------|
| ElectricSQL | String SQL (loose) | `string[]` (loose) | Generic `<T>` on output only | No |
| PowerSync | Via ORM (Kysely/Drizzle) | Via ORM | TS schema definition | No |
| Replicache | Untyped function | N/A | Mutator types only | No |
| Prisma | `WhereInput` objects (strong) | `select: { field: true }` (strong) | `.prisma` schema file | Yes |
| Drizzle | Column references (strong) | Object select (strong) | Schema-as-TS-code | No |
| tRPC | Zod input (strong) | N/A (RPC output) | Zod schemas | No |

## Codebase Findings

### Current Shape Implementation

- `packages/core-rust/src/schema.rs` (line 137-151): `SyncShape` struct with `limit: Option<u32>`. No default, no max.
- `packages/server-rust/src/service/domain/shape.rs` (line 214-216): Limit applied during record scan. Records beyond limit are counted (`total_matches`) but not collected. `has_more: Option<bool>` signals more data exists.
- `packages/core/src/schemas/shape-schemas.ts` (line 24-31): `SyncShapeSchema` with `limit: z.number().int().optional()`.
- `packages/client/src/sync/ShapeManager.ts` (line 20-24): `ShapeSubscribeOptions` with `filter?: PredicateNode`, `fields?: string[]`, `limit?: number`. No type parameter.
- `packages/client/src/ShapeHandle.ts`: `filter: any`, `fields: string[] | undefined`. Loose typing.
- `packages/core-rust/src/messages/base.rs` (line 104-112): `PredicateNode` uses `attribute: Option<String>` and `value: Option<rmpv::Value>`. Fully dynamic, no schema binding.

### Relevant Patterns

- TopGun already uses Zod for schema validation (`@topgunbuild/core` schemas)
- `MapSchema` exists in `core-rust/src/schema.rs` with field definitions and constraints
- The project plans Zod-to-Rust codegen (PROJECT.md: "TypeScript-first schema strategy")
- `PredicateNode` is a recursive expression tree supporting `And`, `Or`, `Eq`, `Gt`, `Lt`, `Contains`, etc.

## Recommendations

### 1. Limits

**Recommended approach:** Server-side configurable max limit with sensible default.

**Design:**
- Add `max_shape_records` to server configuration (default: `10,000`)
- When client sends no `limit`, server applies `max_shape_records` as ceiling
- When client sends `limit > max_shape_records`, server clamps to `max_shape_records`
- Always set `has_more: true` when clamped, enabling client-side pagination awareness
- No change to wire protocol (existing `limit`, `has_more` fields suffice)

**Reasoning:**
- No competing project enforces defaults, but TopGun is a data grid (not just a sync engine). Hazelcast's Near Cache defaults to 10K entries for NATIVE format - a reasonable precedent.
- Without limits, a single `subscribeShape("events", {})` on a million-row table would serialize all records into a single WebSocket message, killing server memory and client performance.
- 10,000 is generous enough for most CRUD apps but prevents runaway subscriptions.
- Server-side enforcement is critical because clients can be malicious or buggy.
- The `has_more` flag already exists in the protocol, so clients can detect truncation.

**Implementation notes:**
- Config field: `ShapeConfig { max_records: u32 }` with `Default` impl returning `10_000`
- Apply in `ShapeService::handle_shape_subscribe()` before the record scan loop
- Log a warning when clamping: helps operators diagnose "missing data" reports
- Future: add cursor-based pagination for shapes (new `SHAPE_PAGE_REQ` message) to let clients page through large result sets

### 2. Type Safety

**Recommended approach:** Phase the type safety improvements. Start with generic output typing (Phase A), add schema-derived filter typing later (Phase B).

**Phase A: Generic output typing (low effort, high value)**

```typescript
// Current (loose)
subscribeShape(mapName: string, options?: ShapeSubscribeOptions): ShapeHandle

// Phase A (typed output)
subscribeShape<T = Record<string, unknown>>(
  mapName: string,
  options?: ShapeSubscribeOptions
): ShapeHandle<T>

// ShapeHandle<T> replaces records: Map<string, any> with records: Map<string, T>
// ShapeUpdate.value becomes T | undefined instead of any
```

This is what ElectricSQL does and it provides immediate value: autocomplete on shape data without requiring codegen.

**Phase B: Schema-derived filter typing (higher effort, requires codegen)**

```typescript
// With Zod schema registration, generate filter types
type UserFilter = {
  status?: string | { eq?: string; in?: string[] };
  age?: number | { gt?: number; lt?: number; gte?: number; lte?: number };
};

type UserFields = 'id' | 'name' | 'email' | 'status' | 'age';

subscribeShape<User>(
  mapName: 'users',
  options?: {
    filter?: PredicateNode; // or UserFilter (sugar that compiles to PredicateNode)
    fields?: UserFields[];
    limit?: number;
  }
): ShapeHandle<User>
```

This is the Prisma/Drizzle approach. It requires:
1. Zod schema definitions for each map (already planned per PROJECT.md)
2. A codegen step or type utility that derives filter types from Zod schemas
3. A `PredicateBuilder` helper that converts typed filter objects to `PredicateNode` trees

**Reasoning:**
- Phase A is a straightforward generic addition. It follows the ElectricSQL pattern and provides 80% of the DX improvement.
- Phase B depends on the schema system (TODO-069) being in place. Prisma-style codegen is the gold standard but requires tooling investment.
- The `PredicateNode` wire format should not change. Typed filters are a client-side compilation layer.
- Drizzle's schema-as-code approach aligns better with TopGun's Zod-first philosophy than Prisma's separate schema file approach.

**Implementation notes for Phase A:**
- Add generic `T` to `ShapeHandle<T>`, `ShapeUpdate<T>`, `ShapeUpdateCallback<T>`
- `records: Map<string, T>` instead of `Map<string, any>`
- `ShapeManager.subscribeShape<T>()` passes through the generic
- No runtime changes needed - this is purely a TypeScript type layer
