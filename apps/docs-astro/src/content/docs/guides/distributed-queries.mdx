# Distributed Queries

TopGun distributes data across 271 partitions spread over all cluster nodes. Most queries can be answered by a single node, but queries that need data from multiple partitions — particularly aggregations — use a **DAG (directed acyclic graph) execution pipeline** to fan work out across the cluster and merge the results.

## When Queries Go Distributed

Not every query needs the full DAG pipeline. TopGun automatically chooses the right execution strategy based on the query type:

- **Simple key lookup** — routed directly to the partition that owns the key. Single-node, lowest latency.
- **Predicate query (no aggregation)** — the coordinator fans out a scan to all nodes, each node filters locally, and the coordinator merges the matching records.
- **GROUP BY / aggregation** — the coordinator compiles a DAG plan. Each node runs a partial scan + aggregate, and the coordinator combines the partial results into the final output.
- **SQL query (no aggregation)** — executes locally on the connected node via Apache DataFusion.
- **SQL query (with aggregation)** — uses the same DAG pipeline as predicate-based GROUP BY queries.

## DAG Architecture

When a query requires distributed aggregation, TopGun compiles it into a DAG plan — a graph of processing steps (vertices) connected by data flow edges.

### Processor Types

| Processor | Role |
|-----------|------|
| **Scan** | Read records from a partition's local storage |
| **Filter** | Apply predicate conditions to discard non-matching records |
| **Project** | Select specific fields from each record |
| **GroupBy (partial)** | Run per-partition aggregation (COUNT, SUM, etc.) |
| **GroupBy (final)** | Combine partial aggregates from all partitions |
| **Sort** | Order results by the requested columns |
| **Limit** | Truncate results to the requested count |
| **Combine** | Merge streams from multiple partitions into one |

### Execution Flow

1. The coordinator compiles the query into a DAG plan describing which processors to run and how data flows between them
2. The plan is split into per-partition sub-plans — each sub-plan contains Scan, Filter, Project, and partial GroupBy processors
3. Sub-plans are dispatched to the nodes that own the target partitions
4. Each node executes its sub-plan locally and streams partial results back to the coordinator
5. The coordinator runs the final Combine, GroupBy (final), Sort, and Limit processors to produce the result

## Partition-Aware Execution

TopGun uses consistent hashing to distribute data across 271 partitions. Each partition is owned by a specific cluster node, with backup copies on other nodes for fault tolerance.

When a distributed query executes:

1. **Partition mapping** — the coordinator determines which nodes own which partitions
2. **Parallel dispatch** — sub-plans are sent to all relevant nodes simultaneously
3. **Local execution** — each node scans only its own partitions, applying filters and partial aggregation locally
4. **Result streaming** — partial results flow back to the coordinator as they become available
5. **Final merge** — the coordinator combines all partial results into the final output

This partition-aware approach means that adding more nodes to the cluster increases query throughput proportionally — each node handles fewer partitions and returns results faster.

## Query Type Comparison

| Query Type | Execution | Latency | Use Case |
|---|---|---|---|
| Key lookup | Single partition | Lowest | Get/set by key |
| Predicate (no agg) | Fan-out scan | Low | Filtered lists, subscriptions |
| GROUP BY / Aggregation | DAG fan-out | Medium | Analytics, summaries |
| SQL (no agg) | Local DataFusion | Low-Medium | Ad-hoc exploration |
| SQL (with agg) | DAG fan-out | Medium | Complex analytics |

### Execution Trace

Here is what happens when this query executes on a 3-node cluster:

1. **Compile** — the coordinator parses the query and builds a DAG plan: Scan -> Filter (`total > 0`) -> GroupBy (partial, group by `status`, aggregate COUNT + SUM + AVG) per partition
2. **Dispatch** — sub-plans are sent to Node A (partitions 0-90), Node B (partitions 91-180), and Node C (partitions 181-270)
3. **Partial aggregation** — each node scans its local `orders` records, filters by `total > 0`, and computes partial aggregates per `status` group
4. **Combine** — the coordinator receives three sets of partial results and merges them: COUNTs are summed, SUMs are summed, AVGs are recomputed from the merged SUM/COUNT
5. **Sort + Limit** — the coordinator sorts by `total_revenue DESC` and takes the top 10 groups
6. **Deliver** — the final result is sent back to the client
