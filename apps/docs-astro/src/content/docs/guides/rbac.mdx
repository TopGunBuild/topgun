# RBAC — role-based access control

TopGun enforces **per-collection, per-operation** access control on the server. A Tower middleware layer checks every client read, write, and remove against a policy store before the operation reaches the domain services. This is shipped and active today — you do **not** need to re-implement authorization in your application code. App-layer claim checks are fine as optional defense-in-depth, but server-side enforcement is the authoritative gate.

## How evaluation works

Authorization runs on every client operation (`read`, `write`, `remove`):

1. **Admin bypass** — a principal whose roles include `admin` is always allowed.
2. **Permissive default** — if the policy store is empty, all operations are allowed, so an unconfigured deployment behaves exactly as before. RBAC "activates" the moment you create your first policy.
3. **Allow-list once active** — with at least one policy present, a map is evaluated against the policies whose pattern matches it. The principal needs a matching `allow` for the action.
4. **Deny-wins** — if any matching policy denies the action, the result is `Deny`, regardless of allow rules.
5. **Default-deny for uncovered maps** — once any policy exists, a non-admin operation against a map with no matching `allow` is denied.

<AlertBox
  variant="warning"
  title="The first policy switches the whole server to allow-list mode"
  text="Because uncovered maps default to deny once any policy exists, adding a single narrow policy can lock non-admin clients out of every other map. Plan your policy set so that each collection your clients use has an explicit allow rule (a broad pattern like '*' works for a baseline)."
/>

## Policy model

A policy is a single rule binding a map pattern + action to an allow/deny effect, with an optional condition:

| Field | Type | Meaning |
|-------|------|---------|
| `id` | string | Stable identifier. Generated as a UUID if omitted on create. |
| `mapPattern` | string (glob) | `*` matches all maps; `users.*` matches `users.profiles`, `users.settings`, etc. |
| `action` | `read` \| `write` \| `remove` \| `all` | The operation governed. `all` covers every action. |
| `effect` | `allow` \| `deny` | Whether a match grants or denies access. |
| `condition` | predicate (optional) | Rule only applies when the condition evaluates true against the record data and auth context. |

## Defining policies (admin API)

Policies are managed over the admin HTTP API. The endpoints require an **admin bearer token** (mint one by logging in with the operator credentials — see [self-hosting](/docs/deploy/self-host)).

```http
POST   /api/admin/policies        # create (or upsert with an explicit id)
GET    /api/admin/policies        # list all policies
DELETE /api/admin/policies/{id}   # delete by id (204 even if absent)
```

Create an allow rule that lets any authenticated user read the `articles` map:

```bash
curl -X POST https://api.example.com/api/admin/policies \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "mapPattern": "articles",
    "action": "read",
    "effect": "allow"
  }'
```

Add a deny rule — deny-wins, so this blocks writes to anything under `admin.*` even if a broader allow exists:

```bash
curl -X POST https://api.example.com/api/admin/policies \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "mapPattern": "admin.*",
    "action": "write",
    "effect": "deny"
  }'
```

## Conditions

A policy can carry a condition that is evaluated against the record `data` and the caller's auth context. Conditions reference `data.<field>` and `auth.id` / `auth.roles`. The simplest way to express one is the `conditionExpr` string, a CEL-like expression parsed server-side:

```bash
# Owners can write only their own records
curl -X POST https://api.example.com/api/admin/policies \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "mapPattern": "documents",
    "action": "write",
    "effect": "allow",
    "conditionExpr": "auth.id == data.ownerId"
  }'
```

More expression examples:

```text
auth.id == data.ownerId
data.role in auth.roles || data.public == true
data.x == 'hello' && (auth.role == 'admin' || auth.role == 'editor')
```

Supported operators: `==`, `!=`, `>`, `<`, `>=`, `<=`, `in`, `contains`, `containsAll`, `containsAny`, `startsWith`, `endsWith`, combined with `&&`, `||`, `!`, and parentheses. Values are strings (`'...'`), numbers, or booleans. If you prefer to pass a structured predicate directly, send a `condition` JSON object instead — it takes precedence over `conditionExpr`.

## Roles

Roles come from the authenticated principal (the JWT validated on the WebSocket handshake — see [Authentication](/docs/guides/authentication)). The `admin` role short-circuits evaluation and is always allowed. Other roles are matched inside conditions via `auth.roles` (e.g. `data.role in auth.roles`).
