# Shapes (Partial Replication)

Shapes let a client subscribe to a **filtered subset** of a map rather than replicating every record. The server evaluates the filter and sends only the records that match, then streams live updates as records enter or leave the shape.

## What Are Shapes

A **shape** is a named, parameterized view of a map:

- **Map name** ‚Äî which map to subscribe to (e.g. `'users'`, `'orders'`)
- **Filter** ‚Äî a `PredicateNode` expression evaluated server-side (e.g. `status == 'active'`)
- **Fields** ‚Äî optional list of fields to include in each record's value (projection)
- **Limit** ‚Äî optional cap on the initial snapshot size

Compare this to `client.map()`, which replicates **all records** in a map. A shape is the right choice when a map is large but the client only cares about a small subset ‚Äî for example, the tasks assigned to the current user, or orders placed in the last hour.

The filter runs **on the server**. The client does not receive records that do not match, and cannot access them through the shape handle.

## ShapeHandle API Reference

`subscribeShape()` returns a `ShapeHandle`. All fields marked **readonly** are set at creation time and do not change.

| Member | Type | Description |
|--------|------|-------------|
| `shapeId` | `string` (readonly) | Auto-generated UUID identifying this subscription on the server. |
| `mapName` | `string` (readonly) | The map this shape is subscribed to. |
| `filter` | `PredicateNode \| undefined` (readonly) | The filter expression this shape was created with. Useful for inspection and reconnect. |
| `fields` | `string[] \| undefined` (readonly) | The field projection this shape was created with. |
| `limit` | `number \| undefined` (readonly) | The record limit this shape was created with. |
| `records` | `Map<string, any>` | Current matching records. Populated from the server snapshot (`SHAPE_RESP`) and updated by live events. |
| `merkleRootHash` | `number` | Merkle root hash from the last server snapshot. Used internally for delta sync on reconnect. |
| `onUpdate(callback)` | `() => void` | Register an update listener. Returns an unsubscribe function that removes only this callback. |
| `unsubscribe()` | `void` | Stop all updates, send `SHAPE_UNSUBSCRIBE` to the server, and clear local state. |
| `isUnsubscribed()` | `boolean` | Returns `true` after `unsubscribe()` has been called. |

## ShapeUpdate Events

Every callback registered with `onUpdate()` receives a `ShapeUpdate` object:

| Field | Type | Description |
|-------|------|-------------|
| `key` | `string` | The record key that changed. |
| `value` | `any \| undefined` | Present for `ENTER` and `UPDATE` events. Absent (`undefined`) for `LEAVE` events. |
| `changeType` | `'ENTER' \| 'UPDATE' \| 'LEAVE'` | The kind of change. |

**When each fires:**

- **`ENTER`** ‚Äî A record now matches the shape's filter. Either a new record was written that matches, or an existing record was updated so that it now satisfies the filter.
- **`UPDATE`** ‚Äî A record that already matches the filter had its value changed. It still satisfies the filter, but some fields changed.
- **`LEAVE`** ‚Äî A record no longer matches the filter. Either the record was deleted, or it was updated so that it no longer satisfies the filter.

## Filter Syntax

Filters are plain JavaScript objects conforming to the `PredicateNode` interface, exported from `@topgunbuild/client`.

**Comparison predicates** ‚Äî match a single field against a value:

```typescript
{ op: 'eq',    attribute: 'status', value: 'active' }
{ op: 'neq',   attribute: 'status', value: 'deleted' }
{ op: 'gt',    attribute: 'age',    value: 18 }
{ op: 'gte',   attribute: 'score',  value: 80 }
{ op: 'lt',    attribute: 'price',  value: 100 }
{ op: 'lte',   attribute: 'rank',   value: 10 }
{ op: 'like',  attribute: 'email',  value: '%@example.com' }
{ op: 'regex', attribute: 'name',   value: '^Alice' }
```

**Logical predicates** ‚Äî combine other predicates using a `children` array:

```typescript
{ op: 'and', children: [pred1, pred2, ...] }
{ op: 'or',  children: [pred1, pred2, ...] }
{ op: 'not', children: [pred1] }
```

## Field Projection

Pass a `fields` array to tell the server to return only those fields in each record's value. This reduces payload size for wide records where the client only needs a few columns.

Fields not listed in the `fields` array are stripped by the server before sending. All records in `shape.records` and all `ShapeUpdate` values will contain only the projected fields.

## Limits

Pass `limit` to cap the number of records in the initial snapshot:

<div className="not-prose">
  <CodeBlock title="Limiting the initial snapshot" language="typescript" code={limitCode} />
</div>

When the server has more matching records than `limit`, the initial `SHAPE_RESP` message includes `hasMore: true`. The `shape.records` map will contain at most `limit` entries.

**Important:** the limit applies only to the initial snapshot. Live `SHAPE_UPDATE` events still arrive for every subsequent mutation that matches the filter, regardless of whether the snapshot was truncated. If you need stable pagination, track the records yourself and apply client-side trimming.

## Offline and Reconnection

Shape subscriptions survive disconnects automatically. The `ShapeHandle` holds the full shape definition (map name, filter, fields, limit) and the `merkleRootHash` of the last received snapshot in memory.

When the client reconnects:

1. `ShapeManager.resubscribeAll()` is called automatically by the sync engine.
2. For each active shape, the engine sends a `SHAPE_SUBSCRIBE` message followed by a `SHAPE_SYNC_INIT` message.
3. `SHAPE_SYNC_INIT` carries the stored `merkleRootHash`, allowing the server to compute a delta ‚Äî only the records that changed since the last sync are sent, not the full snapshot again.
4. The `shape.records` map is patched in-place and `onUpdate` callbacks fire for each delta record.

No user action is required. Reconnection and delta sync are transparent.

## Performance Considerations

**Server memory per shape.** The server maintains a per-shape Merkle tree to support efficient delta sync. Each active subscription consumes a small amount of server memory proportional to the number of matching records. Many shapes on one connection multiply that cost.

**Prefer specific filters.** A shape with `{ op: 'eq', attribute: 'userId', value: currentUserId }` matches far fewer records than a shape with no filter. Narrow filters reduce both server memory and network traffic.

**Use field projection.** If your records have many fields but your UI only needs a subset, pass `fields` to cut payload size. This is especially valuable on mobile connections or for high-frequency update maps.

**Unsubscribe when done.** Call `shape.unsubscribe()` when a component unmounts or a feature is no longer in use. This sends `SHAPE_UNSUBSCRIBE` to the server, releasing the Merkle tree and removing the connection from the server's broadcast set.

**Limit large initial snapshots.** Use `limit` if you only need a bounded view (e.g. "the 50 most recent pending orders"). This avoids transferring the full matching set on connect.
