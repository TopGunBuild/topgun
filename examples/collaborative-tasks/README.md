# Collaborative Task Board

A working example of **Tier 1 adoption** -- adding TopGun real-time features to an existing Express + PostgreSQL application.

This example demonstrates the **hybrid pattern**: REST handles CRUD, PostgreSQL is the source of truth, and TopGun handles real-time collaboration (presence indicators and live task updates). No database replacement needed.

## Architecture

```
 Browser A                   Browser B
 +------------------+       +------------------+
 | Vanilla JS UI    |       | Vanilla JS UI    |
 | - REST calls     |       | - REST calls     |
 | - TopGun WS sub  |       | - TopGun WS sub  |
 +--------+---------+       +--------+---------+
          |                           |
   REST   |    +--- WebSocket --------+
   (CRUD) |    |
          v    v
 +--------+----+--------+
 | Express Server :3000  |
 | +------------------+  |
 | | REST Routes      |  |  <-- CRUD via pg (node-postgres)
 | | (tasks.ts)       |  |
 | +--------+---------+  |
 |          |             |
 |          | publishes   |
 |          v             |
 | +------------------+  |
 | | TopGun Bridge    |  |  <-- Publishes changes to TopGun
 | | (topgun-setup.ts)|  |
 | +--------+---------+  |
 +---------|-------------+
            |
            v
 +---------------------+
 | TopGun Rust Server   |
 | :8080 (WebSocket)    |
 +---------------------+
```

**Data flow:**

1. User creates/updates/deletes a task via the REST API
2. Express route writes to PostgreSQL (source of truth)
3. Route calls `publishTaskUpdate()` which sends to the TopGun topic
4. All connected browsers receive the update via WebSocket subscription
5. Presence is tracked via a TopGun LWW map (`task-presence`)

## Prerequisites

Before running this example, you need:

1. **Node.js 18+** and **pnpm** installed
2. **PostgreSQL** running locally (default: `postgres://postgres:postgres@localhost:5432`)
3. **TopGun Rust server** running in a separate terminal:
   ```bash
   # From the repository root:
   cargo run --release --bin test-server
   ```
4. **Built packages** at the monorepo root:
   ```bash
   # From the repository root:
   pnpm install
   pnpm build
   ```

## Setup

```bash
# 1. Create the database and table
psql -f schema.sql

# 2. (Optional) Copy and edit environment config
cp .env.example .env

# 3. Install dependencies
pnpm install

# 4. Start the app (bundles frontend JS, then starts Express)
pnpm dev
```

Open `http://localhost:3000` in your browser. Open a second tab to see real-time collaboration in action.

## How the Hybrid Pattern Works

This example keeps a clear separation between two concerns:

| Concern | Technology | Files |
|---------|-----------|-------|
| **CRUD persistence** | Express + PostgreSQL (pg) | `server.ts`, `routes/tasks.ts` |
| **Real-time collaboration** | TopGun client + Rust server | `topgun-setup.ts`, `app.js` |

The Express server never imports from `@topgunbuild/server`. It uses the `@topgunbuild/client` package to connect to the Rust TopGun server as a client, just like the browser does.

After each REST mutation (POST, PUT, DELETE), the route calls a TopGun bridge function that publishes the change to a `task-updates` topic. Every browser connected via WebSocket receives this update instantly.

Presence tracking uses a `task-presence` LWW map. Each browser writes its user ID and the task it is viewing. Other browsers read this map to display colored presence dots.

## What to Try

1. **Real-time updates:** Open two browser tabs. Create a task in one tab and watch it appear in the other without refreshing.

2. **Status changes:** Click a status button to move a task between columns. The change propagates to all tabs instantly.

3. **Presence:** Each tab gets a random user ID with a colored dot. You can see who else is connected.

4. **Offline resilience:** Stop the TopGun Rust server. The REST API continues working (tasks are in Postgres). Start the server again and real-time sync resumes automatically.

5. **Inspect the bridge:** Read `src/topgun-setup.ts` to see how REST mutations are bridged to TopGun topics. This is the core integration pattern.

## Key Takeaways

- **No database replacement:** PostgreSQL remains the source of truth for all data
- **Minimal integration surface:** Only 3 functions bridge REST to TopGun (`publishTaskUpdate`, `publishTaskDelete`, `setPresence`)
- **No framework required:** Vanilla HTML/JS demonstrates TopGun works with any frontend stack
- **Separate processes:** The TopGun Rust server runs independently; the Express app is just a client
- **Graceful degradation:** If the TopGun server is down, CRUD still works -- only real-time features are affected
